# Kronos

A personal hours tracker. Single-user. Network-only (no auth). SQLite-backed. Runs as a Docker container behind your own reverse proxy.

- **Backend:** FastAPI + SQLAlchemy 2 + Alembic
- **Frontend:** Jinja2 shell + Alpine.js + Pico CSS + Chart.js (all vendored, no build step)
- **Storage:** single SQLite file (WAL mode) in a Docker volume
- **Deployment target:** LXC container on Proxmox, fronted by NGINX Proxy Manager at `hours.nutello.cc`

## What it tracks

One row per day (`work` / `vacation` / `sick` / `holiday`). Work days have start/end times and any number of breaks. Weekly, monthly, and cumulative surplus/deficit are computed against a configurable daily target (default 8h). Non-work days reduce the target for that day to 0 — the weekly/monthly target adjusts accordingly.

## Quick start — deploy on Dockerusmaximus

1. Copy or clone this repo onto the Docker host (e.g., `/opt/kronos`).
2. Build + start:
   ```sh
   docker compose up -d --build
   ```
   First boot runs `alembic upgrade head` automatically and seeds default settings.
3. Point NGINX Proxy Manager: `hours.nutello.cc` → `http://192.168.68.79:8765`.
4. Add the DNS record in PiHole.
5. Open `https://hours.nutello.cc`. On your Surface, "Install app" from the browser menu pins it to the taskbar (PWA, thanks to `manifest.json`).

### If port 8765 is taken

Set `APP_PORT` in Portainer's stack env or in a `.env` next to `docker-compose.yml`:
```
APP_PORT=8800
```
Both the container and the host-port binding pick this up.

## Local development

Requires Python 3.12+ somewhere on PATH (the Windows Store alias doesn't count — use a real install or WSL).

```sh
make install          # pip install -r requirements-dev.txt
make migrate          # alembic upgrade head
make seed             # ~3 months of realistic sample data
make run              # uvicorn on :8765 with --reload
```

Open http://localhost:8765.

## Tests

```sh
make test             # pytest from repo root; in-memory SQLite, no container needed
make lint             # ruff check + format check
```

Tests cover:
- **Unit:** net-hours math, break-calc conversions, ISO-week and month-boundary helpers, target computation across mixed day types, leap years
- **API:** every endpoint, including validation errors (duplicate date, end ≤ start, break > span, day-type transition rejections, malformed dates)
- **Integration:** full CRUD flows, dashboard recalculation after state changes, week-spanning-two-months math, export round-trips

Run tests inside the image (optional):
```sh
docker build --target test .
```

## Configuration

| Env var | Default | Notes |
|---|---|---|
| `APP_PORT` | `8765` | Internal + published port |
| `TZ` | `Europe/Madrid` | Drives what "today" means in the dashboard |
| `KRONOS_DATA_DIR` | `/app/data` (container) / `./data` (local) | Where `kronos.db` lives |
| `DATABASE_URL` | derived from `KRONOS_DATA_DIR` | Full override for the SQLAlchemy URL |

Runtime-editable config (via `PUT /api/config` or the Settings tab):
- `daily_target_hours` — default 8.0
- `cumulative_start_date` — default 2025-01-01

These live in the `settings` table; the migration seeds defaults on first boot.

## Data & backups

The SQLite file is at **`/app/data/kronos.db`** inside the container, bound to the **`kronos-data`** named volume on the host.

Back up on the host:
```sh
docker run --rm -v kronos-data:/data -v "$PWD":/out alpine \
  sh -c 'cp /data/kronos.db /out/kronos-$(date +%F).db'
```
(WAL is checkpointed by SQLite on connection close; for a tighter guarantee, stop the container before copying.)

## Migrations

```sh
make migrate                           # apply
make revision MSG="add new column"     # new autogenerated migration
```
Alembic runs with `render_as_batch=True` so SQLite-unfriendly `ALTER TABLE` ops (drop column, alter type) work.

## API

| Method & path | Purpose |
|---|---|
| `POST /api/entries` | Create a day entry (with breaks) |
| `GET /api/entries?from=&to=` | List entries, optional date range |
| `GET /api/entries/{date}` | Single entry |
| `PUT /api/entries/{date}` | Full replace (atomic break-set replace) |
| `DELETE /api/entries/{date}` | Delete (cascades breaks) |
| `GET /api/dashboard?today=` | Week + month + cumulative summary |
| `GET /api/analytics/cumulative?as_of=` | Point-in-time surplus/deficit |
| `GET /api/analytics/monthly` | One row per month |
| `GET /api/analytics/records` | Longest day, most surplus month, etc. |
| `GET /api/export.csv` | Download CSV |
| `GET /api/export.json` | Download JSON |
| `GET /api/config` / `PUT /api/config` | Read/write daily target + cumulative start date |
| `GET /healthz` | Container healthcheck |

Interactive docs at `/docs` (FastAPI's Swagger UI).

## Security posture

Designed for a trusted internal network behind WireGuard:
- **No auth layer.** Access control is the network itself.
- **No cookies**, so no CSRF surface to defend.
- **JSON-only mutation API** + Pydantic-strict input validation on every field.
- Container runs as a **non-root user** (`kronos`, uid 1000).
- Uvicorn runs with `--proxy-headers` so the real client IP surfaces in logs when behind NPM.

If you ever expose this outside the VPN, add an auth layer (e.g. Cloudflare Access in front) — the app is not hardened for the open internet.
